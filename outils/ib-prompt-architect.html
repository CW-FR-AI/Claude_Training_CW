IB Prompt Architect.

Sur invocation explicite (@architect), structure un prompt rigoureux paste-ready en 5 sections : diagnostic, questions, setup recommandé, prompt, runbook. Ce Skill ne fait pas le travail — il prépare le prompt que vous lancerez ailleurs.

— Ce qu'il fait

Un méta-skill : il prépare, il n'exécute pas

Architect est un générateur de prompts pour les tâches IB complexes. Vous lui décrivez la tâche que vous voulez automatiser ou rendre reproductible — il produit un prompt structuré en 5 sections que vous copiez-collez ensuite dans un nouveau chat pour l'exécuter.

Les 5 sections produites

SectionContenu
1. Quick DiagnosisDiagnostic de la tâche : nature, complexité, environnement recommandé (chat / Cowork), modèle conseillé.
2. Questions to clarifyLes questions à se poser avant de lancer le prompt — périmètre, sources, format attendu, contraintes.
3. Recommended SetupLe contexte à mettre en place : Project dédié ? Skills à invoquer ? Connecteurs MCP nécessaires ? Documents à charger ?
4. Paste-ready PromptLe prompt complet à copier dans un nouveau chat. Rigoureux, structuré, deal-ready.
5. RunbookComment exécuter : modèle à utiliser, points de contrôle à valider, gestion des erreurs si Claude dérive.

Pourquoi c'est utile

Pour des tâches IB qui reviennent (revue de CIM, préparation d'IC, screening sectoriel), Architect produit un prompt réutilisable que vous stockez dans un Projet. Vous le réutilisez à l'identique sur le mandat suivant. C'est de la capitalisation, pas juste un usage ponctuel.

— Déclenchement

Comment l'invoquer

Skill à invocation explicite — contrairement aux autres Skills, vous devez le citer pour qu'il se déclenche.

Formulations qui déclenchent

  • @architect — la plus directe
  • « Use the architect »
  • « Prompt architect »
  • « IB architect »
  • « Structure-moi un prompt pour [tâche] »

Pattern recommandé

@architect — Je veux structurer un prompt pour préparer un IC interne sur un mandat sell-side. Contexte : cible mid-cap SaaS, secteur RH, géo France. Le prompt doit produire une synthèse à présenter à un partner en 15 minutes.

Plus votre description est précise, plus le prompt produit sera pointu. Décrivez : la tâche, le contexte, l'audience, le livrable attendu, les contraintes spécifiques.

— Cas d'usage

Quand l'utiliser plutôt qu'Enhancer

Cas 1

Tâche récurrente à standardiser

Une routine que vous refaites à chaque mandat (revue d'IM, préparation d'IC, panorama sectoriel). Architect produit le prompt, vous le stockez dans un Projet pour réutilisation.

Cas 2

Tâche complexe à plusieurs étages

Un screening multi-critères qui demande plusieurs passes (filtre 1, validation, enrichissement, scoring). Le prompt structuré clarifie l'orchestration.

Cas 3

Tâche à déléguer

Vous voulez qu'un analyste lance une tâche complexe sans avoir à lui ré-expliquer. Architect produit le prompt, vous le lui transmettez avec le runbook.

Cas 4

Tâche que vous ne ferez pas vous-même

Préparation d'un workflow Cowork ou d'un prompt API pour automatisation. Architect produit la spec exécutable.

— Pièges & limites

À garder en tête

Architect ne fait pas le travail C'est sa caractéristique principale : il prépare le prompt, il n'exécute pas. Si vous voulez juste le résultat directement, n'utilisez pas @architect — laissez l'IB Prompt Enhancer se déclencher automatiquement, ou prompt-ez normalement.
  • Le prompt produit est dense. Il peut faire 30-50 lignes pour des tâches complexes. C'est normal — la qualité vient de la précision du brief.
  • Vous devez lancer le prompt dans un nouveau chat pour avoir un environnement propre. Le prompt fait partie du contexte initial, pas d'un fil long.
  • Stockez les prompts utiles dans un Projet ou un document de référence. C'est la valeur cumulative — chaque prompt produit est réutilisable plusieurs fois.
