— Chapitre 1

Le concept — qu'est-ce qu'un MCP ?

MCP signifie Model Context Protocol. C'est le standard ouvert qui permet à Claude de se connecter à des applications externes pour lire et écrire des données réelles — votre Outlook, votre Drive, une base corporate, un outil de project management.

Le bon modèle mental

Sans MCP, Claude est dans sa bulle : il sait ce que vous lui dites et ce qu'il trouve sur le web public. Avec un MCP connecté, Claude gagne accès à un outil tiers exactement comme s'il avait un compte utilisateur dessus. Il peut lire vos emails Outlook, requêter Pappers, vérifier le calendrier S&P Global, créer une tâche Asana — au moment où vous lui demandez.

MCP vs web search vs upload

  • Web search — Claude cherche sur le web public via un moteur de recherche. Bon pour la news, le contenu publié, l'information générale.
  • Upload de fichier — vous lui donnez un document précis, en pièce jointe ou dans la knowledge base d'un Project. Contenu statique.
  • MCP — Claude requête en direct un outil tiers avec votre authentification. Données vivantes, à jour, parfois privées (votre inbox, votre drive).

Les trois cohabitent. Pour un screening de cibles, vous pouvez avoir : web search (news récentes), Pappers (financiers FR), S&P Global (données cotées), upload d'un brief client. Claude orchestre les sources pour vous.

Permissions et authentification

Chaque utilisateur s'authentifie indépendamment sur chaque connecteur. Les permissions sont les vôtres : si vous donnez à Claude l'accès à votre Outlook, il voit ce que vous voyez — pas plus. Si vous arrêtez le partage, l'accès est coupé.

Point important — connexion par utilisateur Dans un Project partagé, chaque membre utilise ses propres connecteurs MCP avec ses propres authentifications. Si le VP a connecté son Outlook, l'associate doit connecter le sien. Claude ne « partage » pas votre accès Outlook avec le reste de l'équipe.

— Chapitre 3

Comment les invoquer dans une conversation

Trois modes d'invocation, par ordre de robustesse.

Mode 1 — Citation explicite (le plus fiable)

Vous nommez le connecteur dans votre message : « Vérifie sur Pappers l'actionnariat de [société] », « Cherche sur S&P Global les multiples du secteur SaaS européen ». Claude appelle directement le bon outil, pas d'ambiguïté.

C'est le mode à préférer pour les tâches importantes ou critiques. La citation par nom élimine toute incertitude sur le choix de l'outil.

Mode 2 — Description du besoin (automatique)

Vous décrivez ce que vous cherchez sans nommer l'outil : « Trouve-moi les comptes annuels 2023 de [société française] », « Bloque-moi une heure mercredi matin ». Claude choisit le bon connecteur en fonction de votre demande. Marche bien dans 90 % des cas, mais peut se tromper si plusieurs connecteurs sont candidats.

Mode 3 — Via une Skill (le plus puissant)

Certaines Skills Clearwater orchestrent plusieurs connecteurs en une seule invocation. Exemples :

  • Daily Email Review — lit Outlook + croise avec votre calendrier + propose des actions
  • Meeting Finder — interroge plusieurs calendriers via Microsoft 365 pour trouver un créneau commun
  • Clearwater Research Sources — utilise les bons connecteurs (Pappers FR, S&P pour les cotés) selon la nature de la requête

Quand une Skill existe, elle est toujours préférable à l'invocation manuelle — elle encode les bons réflexes (cross-vérification, format de sortie, citation des sources).

Vérification de l'authentification Avant d'utiliser un connecteur pour la première fois, allez dans Paramètres → Connectors et vérifiez qu'il est activé et connecté. Sinon, l'appel échoue silencieusement et Claude vous dit qu'il n'a pas trouvé l'information — alors qu'en réalité, il n'a pas pu requêter.
— Chapitre 4

Cas d'usage IB par connecteur

Les patterns à fort ROI selon le connecteur. À piocher selon vos workflows.

Pappers — sociétés françaises

  • Profil société FR — actionnariat, dirigeants, chiffres clés sur 3 ans
  • Bénéficiaires effectifs — qui contrôle réellement une cible
  • Cartographie groupe — filiales, participations, structure
  • Sirenisateur — retrouver le SIREN exact d'une société à partir d'un nom partiel
  • Décisions de justice — vérifier qu'une cible n'a pas de litige significatif

S&P Global — données cotées et transactions

  • Multiples sectoriels — EV/EBITDA, EV/Sales par secteur et géographie
  • Peers cotés — sociétés comparables avec données financières alignées
  • Transactions M&A historiques — pour benchmarker multiples de transactions précédentes
  • Données macro — taux, change, indices sectoriels

Grata — screener PE et non cotées

  • Longlist cibles par critères — taille, géo, secteur, signaux M&A
  • Identification de PE actifs — quels fonds ont récemment investi dans un segment
  • Recherche d'acquéreurs stratégiques — sociétés non cotées d'une certaine taille dans un segment donné

Microsoft 365 — quotidien Clearwater

  • Briefing du matin — sweep des emails reçus, identification des items qui demandent action (via Skill daily-email-review)
  • Suivi d'un fil mandat — retrouver tous les échanges avec un client / acquéreur sur les 30 derniers jours
  • Recherche de créneau de réunion — trouver une heure commune à 5 personnes sur les 2 prochaines semaines (via Skill meeting-finder)
  • Recherche documentaire SharePoint — retrouver le dernier IM d'un mandat ou les conventions firm-wide
  • Préparation de call client — récupérer le fil d'emails passés avec ce client + agenda du jour

Stacking — combiner plusieurs connecteurs

La vraie puissance vient du stacking. Exemples :

  • Préparation d'un IC interne : Microsoft 365 (notes du dernier IC) + Pappers (mise à jour corporate cible) + S&P Global (multiples actualisés)
  • Suivi univers acquéreurs : Microsoft 365 (fil d'emails par acquéreur) + base interne (statut process) + Grata (signaux récents M&A de chaque acquéreur)
— Chapitre 5

Sécurité, permissions, bonnes pratiques

Activer un connecteur, c'est donner à Claude l'accès à un outil avec vos permissions. Quelques règles à connaître avant d'aller plus loin.

Le principe d'authentification

Chaque connecteur fonctionne avec votre propre authentification (typiquement OAuth). Claude ne voit que ce que vous verriez en vous connectant manuellement. Si vous n'avez pas accès à un dossier SharePoint, Claude n'y aura pas accès non plus.

Permissions par défaut — souvent en lecture seule

La plupart des connecteurs s'installent en lecture seule au démarrage. Pour activer l'écriture (envoyer un email, créer une tâche), vous accordez explicitement la permission supplémentaire. Réflexe : si vous ne comptez pas faire écrire Claude, gardez-le en lecture seule. Moins de risques d'erreurs.

Confidentialité des données qui transitent

Sur Clearwater (compte Team / Enterprise) : les données échangées avec les connecteurs ne sont pas utilisées pour l'entraînement des modèles. Elles transitent par les serveurs Anthropic pour le traitement, puis sont oubliées (politique de rétention zéro pour Team / Enterprise sur demande, sinon courte).

Pour les sujets sous NDA strict, voir la formation Confidentialité & data handling.

Désactiver ce qui ne sert pas

Chaque connecteur activé consomme un peu de contexte (sa description est chargée dans le prompt système). Si vous travaillez sur une session sans besoin de connecteurs, désactivez-les pour gagner du quota et accélérer Claude. Réactivez quand vous en avez besoin.

Bonnes pratiques générales (1) Vérifiez les paramètres de chaque connecteur avant la première utilisation. (2) Pour les actions sensibles (envoi d'email, modification de doc), Claude vous demande confirmation explicite. Ne validez pas en automatique. (3) Si Claude vous propose une action via un connecteur que vous ne maîtrisez pas, refusez et reformulez.
— Chapitre 6

Aller plus loin — MCP custom

Au-delà des connecteurs officiels, Clearwater pourrait à terme déployer ses propres MCP custom — c'est l'horizon stratégique du module.

Pourquoi du MCP custom

Les connecteurs officiels couvrent les outils standard du marché. Mais Clearwater a aussi des outils internes (CRM mandat, base interne d'investisseurs avec historique de contact, tracker de mandats). Un MCP custom permettrait à Claude d'y accéder comme à n'importe quel autre connecteur.

Cas d'usage potentiels

  • CRM mandat Clearwater — récupérer l'historique de contact avec un acquéreur, sa préférence sectorielle, ses tickets typiques
  • Base interne d'investisseurs — qualifier rapidement un nouvel investisseur (fonds, ticket, mandats passés ensemble)
  • Tracker de pipeline interne — récupérer le statut consolidé de tous les mandats actifs

Comment ça se construit

Un MCP custom est un serveur que vous (ou votre IT) déployez, qui implémente le protocole MCP standard. Il expose des « tools » (actions que Claude peut appeler) et des « resources » (données que Claude peut lire). La documentation officielle Anthropic fournit le SDK et des exemples.

Pour Clearwater, ça suppose un projet IT dédié — pas un quick win individuel. Mais le ROI peut être très élevé si plusieurs membres de l'équipe interrogent les mêmes données internes plusieurs fois par jour.

Resources pour creuser

  • Documentation MCP officielle
  • Spécification du protocole
Les connecteurs MCP démultiplient Claude. Un Claude bien câblé sur les bons outils ne fait plus 2 fois plus vite — il fait 10 fois plus vite, parce qu'il économise tous les allers-retours entre applications.
