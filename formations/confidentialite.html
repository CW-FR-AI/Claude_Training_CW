Confidentialité
& data handling.

Les règles à connaître pour utiliser Claude sur des dossiers M&A sensibles sans exposer ce qui ne doit pas l'être. Module destiné à toute l'équipe — particulièrement pour les Partners avant un déploiement systématique.

— Chapitre 1

Le contexte — pourquoi ça compte chez Clearwater

Clearwater travaille essentiellement sur des dossiers M&A sous NDA — sell-side, buy-side, capital advisory. La confidentialité n'est pas une préférence, c'est une obligation contractuelle vis-à-vis du client.

Ce qui est typiquement sensible

  • Identité du client cédé / cible tant que l'opération n'est pas publique
  • Données financières non publiées (BP, comptes pré-publication, marges détaillées par segment)
  • Documents transactionnels — term sheets, NBOs, contrats SPA en négociation
  • Liste des acquéreurs contactés et leur statut dans le process
  • Échanges avec management sur les sujets sensibles (restructurations, départs, litiges)
  • Stratégie de prix et plancher de négociation côté client

Pourquoi Claude introduit des enjeux nouveaux

Avant Claude, le risque de fuite de données passait essentiellement par l'email, le partage de fichiers, ou les conversations physiques. Avec Claude, deux nouveaux vecteurs apparaissent :

  • L'upload — un document chargé dans un chat est traité par l'infrastructure Anthropic
  • Le partage d'artifacts ou de Projects — qui peut, mal géré, donner accès à plus que ce qu'on pense

Les deux nécessitent une vigilance spécifique. Ce module en couvre les règles.

Principe directeur La bonne question n'est pas « est-ce que c'est confidentiel ? » mais « si cette information se retrouvait dans les mains d'un concurrent ou d'un autre acquéreur, est-ce que ce serait un problème ? ». Si oui, applique les règles strictes de ce module.
— Chapitre 2

Ce qui se passe avec vos données chez Anthropic

Comprendre où vont vos données, qui les voit, et comment elles sont traitées. Spoiler : c'est rassurant, mais des règles s'appliquent.

Le plan compte

Clearwater est sur un plan Team ou Enterprise. C'est important parce que le traitement des données diffère selon le plan :

  • Plans Pro et Free (individuels) — les données peuvent, par défaut, être utilisées par Anthropic pour améliorer les modèles. L'utilisateur peut le désactiver dans les paramètres, mais ce n'est pas par défaut.
  • Plans Team et Enterprise — les données ne sont pas utilisées pour l'entraînement des modèles. Rétention limitée et configurable. C'est le cadre dans lequel Clearwater opère.

Le trajet d'une requête

Quand vous envoyez un message à Claude :

  1. Le message + l'historique pertinent + les fichiers joints sont transmis aux serveurs Anthropic (chiffrement TLS pendant le transit)
  2. Le modèle traite la requête et produit une réponse
  3. La conversation est stockée sur les serveurs Anthropic pour que vous puissiez y revenir
  4. Elle reste dans votre compte Clearwater, pas dans une base partagée

Ce qu'Anthropic ne fait pas

  • Ne vend pas vos données à des tiers
  • Ne les utilise pas pour entraîner les modèles (sur plans Team / Enterprise)
  • Ne croise pas les données entre clients (un autre cabinet ne voit pas vos conversations)
  • N'affiche pas de publicité dans Claude

Limites du cloud

Cela dit, vos données sont traitées sur les serveurs d'un tiers (Anthropic). Pour les dossiers où le client interdit explicitement tout traitement cloud — par exemple certaines opérations Defense / Gov, ou des dossiers où la NDA mentionne la non-utilisation d'IA externe — Claude ne devrait pas être utilisé. À vérifier au cas par cas avec le client en début de mandat.

— Chapitre 3

Ce qui est OK et ce qui ne l'est pas

Liste concrète des pratiques à adopter et à éviter, organisée par type de contenu.

Documents financiers d'une cible / client

OK de charger dans un Project Clearwater dédié au mandat : IM committed, comptes annuels publics, BP, présentations management. Ces documents sont déjà partagés avec des acquéreurs via la dataroom — leur présence dans Claude ne crée pas d'exposition nouvelle.

Vigilance sur les documents non encore partagés ou en version pré-IM. Charger un BP qui circulera dans 3 semaines, c'est OK. Charger un BP « inflated » utilisé uniquement en interne pour calibrer la cession, c'est plus délicat.

Documents stratégiques internes Clearwater

OK : notes de positionnement sur un mandat, analyses internes, pitchs envoyés, post-mortem.

OK avec discipline de Project : analyses de pricing, structure de fees, plancher de négociation. À mettre dans un Project Clearwater avec accès limité aux personnes du mandat.

Informations sur les acquéreurs / investisseurs

OK : noms publics, profils corporate, données financières publiques (Pappers, S&P), théories sur leur stratégie d'acquisition.

Vigilance : si un acquéreur vous a communiqué en confidence son ticket plancher, sa stratégie d'enchère, ou son intérêt pour une cible, ne le saisissez pas verbatim dans un chat sans cloisonnement de Project. Information à traiter comme un term sheet.

Personnel / management de la cible

OK : noms, fonctions, parcours public, déclarations publiques.

Pas OK : évaluations personnelles, jugements de performance, informations RH sensibles, situations personnelles. Ces sujets ne devraient jamais finir dans un chat — ils relèvent de la communication orale et de notes papier.

Term sheets et documents en négociation

OK avec cloisonnement : analyse d'un term sheet, comparaison entre plusieurs NBOs reçues, draft de réponse — à condition d'être dans un Project dédié, avec accès strictement limité aux personnes du mandat.

Pas OK : copier-coller dans un chat non-Project ou dans un Project partagé avec des personnes hors mandat. Une fois la conversation indexée dans la mémoire, l'information « fuit » dans le contexte de Claude pour cet utilisateur.

Le test rapide Avant de charger ou de coller quelque chose dans Claude, demandez-vous : « si ce contenu était envoyé en clair par email à tous les membres du Project, est-ce que ce serait OK ? » Si non, n'y mettez pas. Tout ce qui est dans un Project est accessible à tous ses membres.
— Chapitre 4

Mode Incognito & segmentation par Project

Deux mécaniques pour isoler activement ce qui doit l'être.

Le mode Incognito

Disponible depuis le menu chat. Une conversation Incognito :

  • N'est pas sauvegardée dans votre historique
  • N'alimente pas la mémoire long terme de Claude sur vous
  • N'est pas indexée pour la recherche de conversations passées
  • N'est pas accessible via les outils de recherche dans d'autres chats

Cas d'usage typiques :

  • Une réflexion stratégique vraiment confidentielle (positionnement sur un pitch concurrent)
  • Un test de prompt sur un cas qui ne doit pas alimenter la mémoire
  • Une question sensible sans rapport avec un mandat précis

Limites du mode Incognito

  • La conversation est quand même traitée sur les serveurs Anthropic — Incognito n'est pas du chiffrement de bout en bout
  • Vous perdez la conversation à la fermeture — pas de récupération possible
  • Les connecteurs MCP fonctionnent quand même — leurs propres logs s'appliquent (Outlook conserve l'accès même si la conversation est Incognito)

Segmentation par Project

Les Projects sont des silos étanches : un chat dans le Project A ne voit pas les memories ni les conversations du Project B. C'est la principale ligne de défense pour cloisonner les mandats sensibles.

Règles d'application :

  1. Tout mandat avec NDA strict → Project dédié dès le kickoff
  2. Accès au Project limité aux membres du deal — pas le partner sponsor s'il n'y est pas opérationnel
  3. Pour les mandats les plus sensibles, créer un compte d'utilisateur partagé n'est pas une solution — chaque membre opère sous son propre compte avec ses propres permissions
  4. À la clôture du mandat, le Project est archivé (et peut être purgé sur décision)
Combiner Project + Incognito Pour les réflexions ultra-sensibles à l'intérieur d'un mandat actif, ouvrir une conversation Incognito depuis le Project : Claude a accès à la knowledge base et aux custom instructions, mais la conversation n'est pas sauvegardée. Idéal pour les analyses « brouillon stratégique » que vous ne voulez pas voir réapparaître dans un futur chat.
— Chapitre 5

Les pièges spécifiques au partage

C'est ici que beaucoup d'équipes se trompent. Le partage d'un artifact, d'un Project, ou même d'un lien de conversation peut exposer plus que prévu.

Piège 1 — Le partage d'artifact embarque les pièces jointes du chat

Le piège le plus dangereux. Quand vous partagez un artifact (lien Share), les viewers gagnent aussi accès aux pièces jointes et fichiers présents dans la conversation qui l'a créé.

Exemple concret : vous chargez un CIM confidentiel dans un chat, vous demandez à Claude de produire un mini-tracker React de Q&A à partir du CIM, vous partagez le tracker à toute l'équipe pour qu'ils l'utilisent. Toute l'équipe gagne accès au CIM, même si certains membres n'auraient pas dû y avoir accès.

Réflexe à généraliser : avant tout partage d'artifact, vérifier les pièces jointes du chat. Si présence de docs sensibles, recréer l'artifact dans un chat propre sans les pièces jointes.

Piège 2 — Le Project partagé trop largement

Ajouter un viewer à un Project lui donne accès à toute la knowledge base, toutes les custom instructions, tous les connecteurs MCP scopés au Project. Pour un partner sponsor qui ne fait que jeter un œil, c'est trop d'accès si la knowledge base contient des analyses sensibles.

Réflexe : pour les Projects mandat, limiter strictement aux personnes opérationnellement impliquées. Le partner sponsor peut être briefé hors Project.

Piège 3 — Le lien public Publish

Sur les plans individuels (Pro / Max), un artifact peut être « publié » via lien public — anyone with the link peut le voir, sans authentification. Mode disponible uniquement sur les plans individuels.

Chez Clearwater (Team / Enterprise), ce mode n'est pas activable. Le seul mode de partage est le mode Share organisation — restreint aux membres Clearwater authentifiés. Bien.

Piège 4 — La mémoire long terme qui « fuit » entre conversations hors Project

Hors Project, toutes vos conversations partagent la même mémoire long terme. Si vous avez analysé un term sheet sensible hors Project lundi, mardi votre conversation sur un sujet sans rapport peut avoir Claude qui « se souvient » du term sheet. Ce n'est pas une fuite externe, mais une pollution de contexte interne.

Réflexe : tout sujet sensible va dans un Project. Hors Project = sujets non-sensibles uniquement.

Piège 5 — La conversation longue qui agrège trop d'infos

Une conversation qui s'étale sur 50 messages sur un même mandat peut concentrer une quantité d'informations sensibles disproportionnée. Si quelqu'un gagne accès à cette conversation (partage maladroit, compte compromis), l'exposition est massive.

Réflexe : segmenter les conversations par sujet. Démarrer un nouveau chat à chaque nouveau sujet. Limiter naturellement la concentration d'informations.

— Chapitre 6

Règles pour les Partners

Pour les Partners qui consultent occasionnellement Claude ou pilotent l'adoption au niveau firme, voici les règles à retenir.

Trois choses à savoir absolument

  1. Clearwater est sur Team / Enterprise. Vos données ne servent pas à l'entraînement d'Anthropic et restent confidentielles. Vous pouvez utiliser Claude sur du contenu mandat sans crainte première.
  2. Les Projects sont des silos. Un mandat sous NDA strict doit avoir son propre Project, avec accès limité. Pas de « catch-all » avec tous les mandats mélangés.
  3. Avant de partager un artifact ou un Project, vérifiez ce qu'il contient. Surtout les pièces jointes — qui « voyagent » avec le partage.

Cinq do

  • Utiliser le mode Incognito pour les réflexions sensibles qui ne doivent pas s'inscrire dans la mémoire long terme
  • Démarrer chaque mandat dans un Project dédié dès le kickoff, pas après quelques semaines d'usage informel
  • Vérifier la liste des membres d'un Project avant d'y déposer de nouveaux documents sensibles
  • Préférer la citation orale aux uploads pour les informations vraiment sensibles (la conversation orale n'est pas indexée)
  • En cas de doute, consulter le VP responsable du déploiement Claude chez Clearwater avant un usage à risque

Cinq don't

  • Ne pas charger de documents sensibles dans un chat hors Project
  • Ne pas partager un artifact sans avoir vérifié les pièces jointes du chat qui l'a créé
  • Ne pas ajouter de membres à un Project sans réfléchir à ce qu'ils gagnent comme accès
  • Ne pas activer de connecteurs MCP sur un compte personnel utilisé pour Clearwater (rester strictement dans le compte Team / Enterprise)
  • Ne pas réutiliser un Project mandat pour un autre mandat — créer un nouveau Project propre

En cas de doute

Le réflexe le plus sûr : ne pas charger et demander à un collègue plus expérimenté. La perte de temps est négligeable comparée au risque de fuite.

Claude est puissant parce qu'il a accès à beaucoup de contexte. Le revers, c'est qu'il a aussi accès à beaucoup de contexte. La discipline de cloisonnement n'est pas un frein — c'est le prix à payer pour utiliser sereinement l'outil dans un environnement IB.
