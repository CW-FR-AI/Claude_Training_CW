— Chapitre 1

Le concept — qu'est-ce qu'un Skill ?

Un Skill, c'est un module de compétence packagé qui dit à Claude « quand telle situation se présente, applique tel comportement ». Une fois installé, il se déclenche tout seul, sans que l'utilisateur ait à le savoir.

Le bon modèle mental

Pense à un Skill comme un collègue ultra-spécialisé qui se réveille quand sa spécialité est sollicitée. Quand quelqu'un dit « peux-tu améliorer ce mail ? », notre collègue « Email Enhancer » lève la main et prend la suite. Quand quelqu'un dit « fais-moi un profil société », c'est « Clearwater Profile » qui s'y colle. Tu n'as pas besoin de l'invoquer explicitement — la description du Skill contient les triggers, et Claude fait le reste.

Skill ≠ prompt ≠ outil

  • Prompt — ce que tu tapes dans le chat à chaque fois. Personnel, jetable.
  • Skill — un comportement installé, déclenché par mots-clés ou contexte. Réutilisable par toute l'équipe, sans avoir besoin de connaître la formulation magique.
  • Outil / MCP — une capacité externe (lire ton calendrier, requêter Pappers). Couvert dans la formation Connecteurs & MCP.

Deux types de Skills, deux ROI différents

Skills de production — encodent un standard d'écriture ou de mise en forme. Exemples Clearwater : Email Enhancer, Slide Text Enhancer, Clearwater Profile. Le ROI est mesurable : moins de temps de relecture, moins de variance entre rédacteurs.

Skills de process — encodent un workflow complet, pas juste un livrable. Exemple : un Skill « Kickoff sell-side » qui, à chaque nouveau mandat, génère la checklist, les templates, et la structure de Project. Le ROI est démultiplié parce qu'on capture une routine entière, pas juste une étape.

Le seuil de packaging Une bonne règle : si tu refais une routine à l'identique 3 fois par mois, c'est un candidat skill. Sinon, garde-la en prompt personnel — le coût de maintenance d'un Skill non utilisé est supérieur à son bénéfice.

Quand chaque Skill se déclenche

Email Enhancer

Phrases qui déclenchent : « peux-tu améliorer cet email », « polish ce mail », « corrige le ton », « rends-le plus pro », « réécris en plus formel ». Aussi sur des verbes seuls : « fix this email », « clean up ». Marche en FR et EN. Préserve toujours l'intention et le degré de formalité du brouillon — il n'invente pas un nouveau ton.

Slide Text Enhancer

Phrases qui déclenchent : « améliore ce slide », « polish ces bullets », « corrige ce titre », « tagline meilleure ». Applique les règles Clearwater dures : pas de « leading », « cornerstone », « innovative », « strong growth », « best-in-class ». Si le brouillon en contient, ils sont reformulés en langage factuel.

IB Prompt Enhancer

Skill silencieuse. Sur toute demande IB casuelle (« fais-moi un panorama du secteur X », « long list pour cible Y »), elle réécrit en interne le prompt en version rigoureuse (cadrage, sources, format de sortie, contraintes) puis exécute. L'utilisateur ne voit pas le prompt enhancé — il voit le résultat de qualité supérieure.

IB Prompt Architect

S'invoque explicitement avec @architect. Quand tu travailles sur une tâche IB complexe et que tu veux un prompt rigoureux réutilisable, l'architect produit 5 sections : Quick Diagnosis, Questions to clarify, Recommended Setup, Paste-ready Prompt, Runbook. Tu copies le prompt produit dans un nouveau chat et tu obtiens un livrable de qualité élevée. C'est un méta-skill : il prépare, il n'exécute pas.

Clearwater Research Sources

Skill silencieuse. Sur toute recherche M&A / IB (cible, fonds, deal, sectoriel, news), elle dirige Claude vers la liste curée de sources Clearwater organisée en 7 catégories : advisory firms, strategy consultants, macro/regulatory bodies, business press, trade press, federations, PE/debt/investor ecosystem. Citations en footnote + bibliography conformes au format banque d'affaires.

Important — les Skills sont déjà installés Tu n'as pas à les invoquer, les installer, ou les configurer. Tu écris naturellement ta requête, ils se déclenchent automatiquement (sauf Architect qui nécessite @architect). Si un Skill ne se déclenche pas alors que tu pensais qu'il devrait, c'est probablement que ta formulation n'a pas matché les triggers — reformule plus explicitement.
— Chapitre 3

Anatomie d'un Skill

Un Skill, c'est un dossier avec un fichier SKILL.md à la racine et, optionnellement, des ressources annexes. Le SKILL.md est divisé en deux parties qui jouent des rôles très différents.

Le frontmatter YAML — le déclencheur

Au début du SKILL.md, entre deux séparateurs ---, on trouve le frontmatter. Deux champs critiques :

  • name — identifiant lisible (ex. « Email Enhancer »). Sert à l'affichage et au logging.
  • description — c'est ici que tout se joue. C'est la seule chose que Claude lit à chaque conversation pour décider si le Skill est pertinent. Limite stricte de 1024 caractères.

Si la description est mal écrite, le Skill peut être parfait dans son corps mais ne jamais se déclencher. À l'inverse, une description trop large fait que le Skill se déclenche partout, y compris quand ce n'est pas pertinent.

Le corps en markdown — l'instruction opérationnelle

Sous le frontmatter, le corps du SKILL.md contient les instructions que Claude reçoit une fois le Skill déclenché. C'est là que tu mets :

  • La structure exacte attendue de l'output
  • Les conventions de format (ton, registre, longueur)
  • Les sections obligatoires vs optionnelles
  • Des exemples positifs courts et un anti-exemple
  • Les références à des ressources externes (templates, scripts)

Heuristique : 200 lignes maximum pour une skill de drafting. Au-delà, il y a du gras.

Ressources annexes

Pour les Skills qui ont besoin de templates, scripts ou références, le pattern propre c'est de séparer :

  • SKILL.md — la logique principale, courte et lisible
  • templates/ — fichiers exemples ou structures cibles
  • references/ — documentation, glossaires, exemples
  • scripts/ — code helper (Python, bash) que le Skill peut exécuter

Le SKILL.md fait référence à ces fichiers (« utilise le template dans templates/profile.docx ») plutôt que de les inliner. Ça garde la logique séparée des ressources.

Le format .skill

Un Skill packagé est livré en archive .skill — techniquement un ZIP qui contient le dossier du Skill. C'est ce format qui s'installe dans l'environnement Claude. Sur Clearwater, l'IT installe les Skills officiels pour toute l'équipe à partir des fichiers .skill.

— Chapitre 4

Les 5 erreurs classiques de conception

Une grande partie des Skills mal conçues échouent pour les mêmes raisons. Connaître ces pièges fait gagner des heures.

Erreur 1 — Description trop générique

Une description qui dit « Use when the user wants to write something » se déclenche partout et perd sa valeur. Une bonne description est dense en triggers spécifiques : phrases-trigger exactes (entre guillemets), langues couvertes, contextes typiques. Plus c'est concret, mieux le Skill se déclenche au bon moment.

Erreur 2 — Description en mode notice plutôt que trigger

Beaucoup de SKILL.md commencent par décrire ce que fait le Skill (« This skill creates company profiles… »). Ça n'aide pas Claude à décider quand l'utiliser. Ce qui aide, c'est les conditions de déclenchement.

Pattern qui fonctionne :

[Une phrase sur ce que produit le Skill]. Use this skill whenever [trigger 1], [trigger 2], [trigger 3]. Triggers also on: "[exact phrase 1]", "[exact phrase 2]". Use even if the user just says [casual phrase] — any request to [intent] should use this skill.

Erreur 3 — Pas de signaux négatifs

Un Skill mature inclut ce qu'il n'est PAS. Sans ça, des faux positifs apparaissent : le Skill se déclenche sur des requêtes qui ressemblent mais ne sont pas pertinentes. Ajoute systématiquement une ligne « Do NOT use when… ».

Exemple : Email Enhancer devrait avoir « Do NOT use for drafting an email from scratch — only for improving an existing draft ». Sinon il se déclenche sur « écris-moi un email à X », ce qui n'est pas son rôle.

Erreur 4 — Corps de skill trop bavard

Le corps du SKILL.md doit être opérationnel, pas pédagogique. Pas besoin de l'historique du Skill, du « pourquoi » il existe, ou de motivation générale. Claude a besoin de savoir quoi produire et comment, pas pourquoi tu as créé ce Skill.

Erreur 5 — Pas de séparation skill / ressources

Tout inliner dans le SKILL.md rend le fichier illisible et fragilise les mises à jour. Pour toute ressource volumineuse (template, exemple long, glossaire), un fichier séparé est la bonne approche. Le SKILL.md reste sous 200 lignes, les ressources vivent à côté.

Le test décisif d'une description Avant de finaliser un Skill, écris 10 phrases que tu pourrais taper dans Claude. Combien devraient déclencher le Skill ? Combien sont effectivement couvertes par la description ? Si le ratio est faible, enrichis les triggers. Si trop large, ajoute des signaux négatifs.
— Chapitre 5

Créer et déployer un nouveau Skill

Pour un VP ou un associate Clearwater qui veut packager une routine personnelle en Skill équipe, voici le parcours pas à pas.

Étape 1 — Valider que c'est bien un Skill

Avant tout, vérifie que ta routine remplit les trois critères :

  • Tu la refais à l'identique plusieurs fois par mois (sinon ce n'est qu'un prompt)
  • D'autres personnes de l'équipe en bénéficieraient (sinon garde-la en prompt personnel)
  • Tu as une norme claire de ce qu'est un bon output (sinon il faut d'abord cadrer la norme)

Étape 2 — Écrire le brouillon

Le moyen le plus rapide : ouvre une conversation avec Claude, décris la routine que tu veux packager (avec exemples d'input et d'output attendus), et demande à Claude de te générer un SKILL.md draft. Tu obtiens en 2 minutes une base solide à itérer dessus.

Étape 3 — Tester

Avant de packager officiellement, teste en mode « custom instruction » ou en collant le contenu en début de chat. Vérifie que :

  • Le Skill produit le bon output sur 3 cas réels différents
  • Il ne dérive pas (variance de format ou de ton entre runs)
  • Il refuse correctement de se déclencher sur des cas hors scope

Étape 4 — Packager

Structure le dossier proprement :

mon-skill/
├── SKILL.md          (frontmatter + corps, <200 lignes)
├── templates/        (optionnel, exemples de sortie)
├── references/       (optionnel, glossaires)
└── scripts/          (optionnel, helpers)

Zippe le dossier et renomme l'extension en .skill. C'est le fichier que tu transmets à l'IT pour installation.

Étape 5 — Déployer à l'équipe

Trois options selon la portée :

  • Skill personnel — installé sur ton compte uniquement, pour test
  • Skill Project — installé sur un Project spécifique, partagé avec les membres du Project
  • Skill équipe Clearwater — installé au niveau de l'organisation, disponible pour tous

Pour les Skills équipe, l'installation passe par l'IT (administrateur du compte Team / Enterprise). Documentation à fournir : nom, description courte, exemples d'usage, mainteneur.

Astuce — contrainte technique du frontmatter Le champ description du SKILL.md a une limite stricte de 1024 caractères. Au-delà, le Skill ne charge pas. Tiens-t'en à ça, et fais tenir tous tes triggers + signaux négatifs dans cet espace.
— Chapitre 6

Cas d'usage à fort ROI

Au-delà des 5 Skills déjà déployés, voici les zones où l'investissement dans un nouveau Skill rapporte le plus chez Clearwater.

Skills de process pour les routines mandat

Idée 1

Sell-side mandate kickoff

À chaque nouveau mandat, génère la checklist structurée (NDA, IM brief, calendrier process, univers initial, kickoff team), produit les templates initiaux, met en place le Project Claude.

Idée 2

Q&A consolidation

Agrège des questions de plusieurs investisseurs, déduplique, classe par thème, priorise, drafte les réponses dans le ton attendu. Routine très récurrente en phase de Q&A.

Idée 3

Buyer outreach drafting

Pour un univers d'acquéreurs, produit les emails first-touch dans la bonne langue, le bon registre, avec la bonne mention du process et des prochaines étapes.

Idée 4

Investor update

Update hebdo standardisé à envoyer au client sell-side : statut process, courbe d'engagement, prochaines milestones, sujets d'attention. Format identique semaine après semaine.

Skills de production complémentaires

  • Teaser drafting — première version de teaser au format Clearwater, à partir d'un brief client de 2-3 pages
  • Management presentation outline — squelette type d'une management presentation, segmenté par secteur
  • NBO analysis — grille d'analyse d'une offre reçue (prix, structure, conditions, financements, calendrier) en format banker
  • Pitch postmortem — debrief structuré après un pitch perdu pour capitaliser sur les retours

Skills d'orchestration (les plus puissants)

Une Skill peut aussi orchestrer un workflow complet. Exemples ambitieux :

  • Weekly mandate review — pour chaque mandat actif, lit l'inbox + le calendrier + le Project Claude, produit une synthèse hebdo des points ouverts, des risques émergents, des décisions à prendre
  • Pipeline tracker auto-update — récupère les statuts de tous les mandats, met à jour un tableau consolidé, identifie les écarts de capacité

Ces Skills d'orchestration combinent plusieurs MCP (Outlook, Calendrier, Project knowledge) et délivrent une valeur très supérieure aux Skills atomiques.

Un bon Skill se reconnaît à ce qu'on ne le remarque plus. Quand tout le monde dans l'équipe produit des emails impeccables sans plus penser à l'« Email Enhancer », c'est qu'il a réussi.
